# LocalWriter-Experimental: LLM Development Context

Этот документ предназначен для ввода в контекст новой LLM, которая будет работать с проектом `LocalWriter-Experimental`. Он содержит подробное описание архитектуры, целей, проблем и неочевидных технических решений, принятых в ходе разработки. Читайте его ВНИМАТЕЛЬНО, прежде чем предлагать архитектурные изменения или рефакторинг.

## 🎯 1. Цель проекта и суть продукта

**Суть продукта:** 
Это LibreOffice Writer Extension (расширение) на Python, которое привносит "AI-форматирование" в локальные документы. Пользователь выделяет кусок неформатированного текста (сырого), жмет кнопку "Apply Template", и система автоматически применяет корпоративные стили (например, "Heading 1", "Normal", "Title") к нужным абзацам.

**Главные фичи:**
1. **100% Локальность:** Работает через локальную Ollama. Данные никуда не уходят.
2. **RAG (Retrieval-Augmented Generation) для стилей:** Мы не захардкоживаем стили. Система парсит эталонные `.docx` (корпоративные шаблоны), сохраняет их структуру (какие стили к какому типу текста применяются) в векторную базу (ChromaDB). При форматировании пользовательского текста, RAG ищет похожие контексты в эталонах и говорит LLM: "Смотри, вот в эталоне 'Договор' оформлено стилем Title. Оформи выделенный текст так же, используя только разрешенные стили".
3. **Бесшовность:** UNO API LibreOffice используется для применения стилей *на месте*, без поломки остального документа (картинки, колонтитулы остаются целыми).

## 🏗️ 2. Архитектура системы

Система состоит из двух независимых частей, которые общаются по HTTP.

### Часть A: The LibreOffice Extension (`/extension`)
Это клиентская часть. Запускается ВО ВНУТРЕННЕМ Python-окружении LibreOffice (UNO).
- **Главное ограничение:** LibreOffice использует встроенный Python (встроен в ОС или в саму LO). В нем **НЕТ** сторонних библиотек (нет `requests`, нет `pydantic`). Использовать можно только стандартную библиотеку Python (`urllib`, `json`) и `uno`.
- **`main.py`:** Точка входа расширения. Рисует UI диалоги (Settings), обрабатывает кнопки тулбара, взаимодействует с текущим документом через UNO (`XSCRIPTCONTEXT`).
- **`client.py`:** Вынесенный HTTP-клиент (на базе `urllib`). Все запросы к Бэкенду и к Ollama идут через него. Был вынесен отдельно, чтобы тестовые скрипты (`tests/`) могли использовать **точно ту же логику общения по сети**, что и само расширение. 
- **`uno_formatter.py`:** Сердце взаимодействия с LO. Принимает JSON-список абзацев и стилей и физически применяет их к выделенному тексту в отрытом документе. 
- **`tracer.py`:** Логирование в `/tmp/localwriter.log`.

### Часть B: The FastAPI Backend (`/backend/app`)
Это "Умный Прокси" и "Умный RAG-движок" между расширением и Ollama. Запускается как отдельный сервис.
- **Ограничение:** Бэкенд ничего не знает об UNO. Он работает только с файлами `.docx` и сырым текстом.
- **`api/endpoints.py`:** Принимает запросы от `client.py`.
    - `POST /v1/completions`: Это **кастомный** endpoint, который *притворяется* Ollama API для расширения, но под капотом делает ОЧЕНЬ много:
        1. Бьет длинный текст на батчи (куски).
        2. Ходит в `rag_engine` за контекстом (какие стили есть в похожем шаблоне).
        3. Формирует сложный System Prompt с RAG-контекстом и списком разрешенных стилей.
        4. Отправляет составной промпт в реальную Ollama (`ollama_client.py`).
        5. Дожидается JSON (где каждому абзацу назначен стиль).
        6. **Semantic Mapper:** (Очень важно!) Бэкенд жестко обогащает ответ LLM точными параметрами из базы (шрифт, размер и т.д.), чтобы `uno_formatter` не упал из-за галлюцинаций LLM.
        7. Возвращает готовый структурированный JSON-список клиенту.
    - `POST /api/ingest`: Эндпоинт для загрузки `.docx` файлов в RAG-базу.
    - `GET /api/tags`: Проксирует вызов в Ollama для списка моделей (используется в Settings UI).
- **`services/rag_engine.py`:** Управляет векторной базой `ChromaDB`. Использует `SentenceTransformer`. Паттерн *Singleton*, потокобезопасная инициализация.
- **`services/style_extractor.py`**: Ультра-сложный парсер `python-docx` (не `lxml` напрямую). Его задача: вытащить ВСЮ визуальную и смысловую разметку из эталонного `.docx` для RAG. Обрабатывает "Direct Formatting" (прямое форматирование) и "Style Hierarchy" (наследование стилей в Word).

## ⚡ 4. Критические технические решения (СЛОЖНОСТИ и ЛОВУШКИ)

НОВАЯ LLM, ПЕРЕД ТЕМ КАК ПИСАТЬ КОД, ПРОЧТИ ЭТО:

### 1. Тестирование (E2E) - Самая сложная часть
Мы потратили много времени, чтобы сделать `test_formatting_quality.py` максимально приближенным к реальности.
- **ЛОВУШКА:** Нельзя импортировать `main.py` или `uno_formatter.py` в тесты, так как тесты гоняются через `poetry` (вне LibreOffice), а у него нет доступа к `uno`.
- **РЕШЕНИЕ:** Поэтому мы вытащили логику общения по сети в `extension/client.py`. Тест импортирует `client.py`, имитируя поведение расширения "как черный ящик".
- **ПРОЦЕСС ТЕСТА:**
    1. **INGEST:** Заливает 200+ эталонных .docx файлов в RAG. (ЛОВУШКА: При запуске 4 worker-ов, `ChromaDB` со SQLite может блокироваться - `database is locked`, приводя к ERR 500 из `endpoints.py`. Ingest лучше делать в 1 поток).
    2. **GROUND TRUTH (GT):** Бэкенд через `style_extractor.py` парсит структуру `.docx` (идеальный вариант разметки). Тест скачивает ее через `/api/extract_ground_truth`.
    3. **LLM CONTEST:** Тест берет голый текст из `.docx`, прогоняет через `call_apply_template` в `client.py` и сравнивает JSON-ответ (какой стиль назначила LLM) с Ground Truth.

### 2. Зависимости (Poetry)
- Всегда используем `poetry` для управления зависимостями в Python бэкенда (`backend/`).
- Для запуска тестов: `poetry run python tests/test_formatting_quality.py`.
- Для сервера: `poetry run uvicorn app.main:app --host 0.0.0.0 --port 8323 --reload`.
- **Строгое правило:** На клиенте (в `extension/`) использование сторонних библиотек строго ограничено модулями из базовой поставки Python (LibreOffice).

### 3. Парсинг ответов LLM и "The Catch-All Fallback"
- **Structured Outputs (JSON Schema):** Чтобы решить проблему галлюцинаций LLM и потерю ключей `id`, мы больше не используем регулярки для парсинга ответов. При вызове Ollama API `endpoints.py` передает строгую `JSON Schema` в параметре `format`. Модель физически вынуждена возвращать массив объектов `[{"id": 1, "style_name": "Normal"}]`.
- **Catch-All Fallback:** Если LLM всё же "теряет" параграфы из переданного батча (возвращает меньше ID, чем было отправлено), бэкенд перед отправкой финального NDJSON-потока находит недостающие ID и принудительно возвращает для них дефолтный стиль (например, `Normal`). Это гарантирует `100% Coverage` и защищает документ от поломки форматирования.

### 4. UNO Formatter: Иллюзия "голого" стиля и создание на лету
- RAG может предложить стиль, которого нет в *текущем* пустом документе (например, `Corp_Contract_Heading`).
- Ранее макрос пытался применить его, падал, и форматировал абзац только через Direct Formatting (сохраняя жирность/цвет из базы, но теряя саму абзацную структуру).
- **Решение:** В `uno_formatter.py` реализована функция `ensure_style_exists`. Перед применением стиля макрос проверяет его наличие. Если стиля нет, он создает новый `ParagraphStyle` на базе `Standard` и "накатывает" в него все визуальные параметры (`bold`, `font_family`, `font_size`) из `Semantic Mapper`.

### 5. Race Condition и Асинхронные Закладки
- В гибридном конвейере ответы от бэкенда прилетают вразнобой: эвристики отвечают за 50мс (ID 1, 5), векторный FastTrack за 200мс (ID 2), LLM за 5 секунд (ID 3, 4).
- Если бы макрос применял стили подряд на клиенте — текст бы перемешался.
- **Индивидуальные закладки:** Чтобы решить это, перед отправкой батча на сервер `main.py` не собирает пустые параграфы (фильтр `strip() == ""`), а для каждого полезного абзаца расставляет свою уникальную закладку вида `LW_AI_Target_173931..._p{ID}`.
- При получении пакета JSON из HTTP стрима, макрос парсит `id` и применяет стиль строго к закладке `_p{ID}`. В конце сессии мусорные закладки удаляются.

### 6. Сетевые таймауты и XTimer Fallback
- Таймаут в `check_connection` для `/api/tags` увеличен.
- **XTimer Fallback:** Интерфейс `com.sun.star.awt.XTimer` недоступен в некоторых сборках LibreOffice (возвращает `NoneType`). Для таких конфигураций реализован fallback-механизм: запускается фоновый `threading.Thread(daemon=True)`, который опрашивает NDJSON очередь ответов раз в 100мс (имитируя поведение таймера UI потока).

### 7. Русский язык
- Все комментарии, планы, и общение с пользователем должны быть строго на русском языке.

## 🧪 5. Правила написания и запуска тестов (Strict Testing Guidelines)

**ВНИМАНИЕ LLM:** Этот проект требует 100% объективности. Мы пишем B2B-инструмент, а не академический прототип. "Зеленые тесты" без реальной проверки функционала считаются критической ошибкой. При работе с тестами (папка `tests/`) строго соблюдайте следующие правила:

### 1. Тестируй ПРОД, а не моки (Zero-Mock Policy для E2E)

* **Запрещено** мокировать ответы сервера или внутреннюю логику парсинга, если тест заявлен как интеграционный или E2E.
* Скрипт `test_formatting_quality.py` **обязан** импортировать боевой клиент `extension/client.py` и использовать его функции (`call_apply_template_ndjson` и т.д.) точно так же, как это делает LibreOffice. Мы берем реальный вывод продакшен-скрипта и оцениваем именно его.

### 2. Смерть за "False Positives" (Никаких тихих падений)

* Если тест ловит критическую ошибку (`Connection refused`, `TimeoutError`, HTTP 500) — он **не должен** писать `pass` или завершаться с `exit code 0`.
* Если итоговый скор (Coverage / CMS) равен `0.0%` — это провал.
* **Правило:** Любой провал бизнес-логики или сетевой инфраструктуры должен немедленно вызывать `sys.exit(1)`. Bash-скрипты автоматизации полагаются на коды возврата (Exit Codes), чтобы прервать пайплайн.

### 3. Инфраструктура важнее скрипта

* E2E тесты не могут работать в вакууме. Прежде чем запускать `test_formatting_quality.py`, bash-скрипт или тестировщик **обязан**:
1. Поднять боевой бэкенд в фоне (`poetry run uvicorn app.main:app --port 8323 &`).
2. Сделать Healthcheck (например, `curl -f http://localhost:8323/api/tags` в цикле `while`), чтобы убедиться, что сервер реально готов принимать запросы.
3. Только после этого запускать тесты.


* Оставляйте сервер в запущенном состоянии (в фоне) после прохождения тестов, если пользователь собирается проводить ручное тестирование из LibreOffice.

### 4. Тестирование краевых случаев (Edge Cases)

* **NDJSON и Стриминг:** Обязательно тестируйте обрывы сети на середине потока и "мусорные" Heartbeat-строки (пустые `\n`). Парсер на клиенте должен молча игнорировать мусор и не падать с `JSONDecodeError`.
* **Блокировки баз данных:** Инжест документов (Ingest) при тестировании должен воспроизводить реальную нагрузку (например, конкурентную отправку), чтобы проверять надежность `FileLock` для ChromaDB.
